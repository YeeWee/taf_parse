## 2026-03-16: 北美报文能见度格式支持

### 问题描述

无法正确解析北美 TAF 报文中的带分数能见度格式（如 `1 1/2SM`）。

**示例报文**:
```
TAF CYYZ 150540Z 1506/1612 08005KT P6SM FEW050 OVC110
FM150800 07008KT P6SM FEW015 OVC060
FM151000 10010G20KT P6SM FEW020 OVC040
TEMPO 1510/1511 4SM -SN BKN020 OVC040
FM151100 10012G22KT 1 1/2SM -SN BKN008 OVC012
FM151500 10015G25KT 2SM -SN BKN008 OVC012
```

**问题表现**:
1. `1 1/2SM` 格式无法识别 - 被分割为 `["1", "1/2SM"]` 两个 token
2. FM 变化组（如 `FM150800`）被跳过，无法解析
3. 北美报文中的 `P6SM`、`1/2SM` 等格式支持不完整

**支持的能见度格式**:
- `xSM` - 整数英里（如 `6SM`、`P6SM`）
- `x/ySM` - 分数英里（如 `1/2SM`、`3/4SM`）
- `z x/ySM` - 带整数部分的分数（如 `1 1/2SM`、`2 1/4SM`）

### 根本原因

1. **行过滤逻辑遗漏 FMddhhmm 格式**
   - `parse_taf()` 中的行过滤只识别 `^[A-Z]{4}\s` 和 `^(TEMPO|INTER|BECMG|PROB\d{2})\s`
   - `FM150800` 格式（FM 后跟 6 位数字）不被识别，整行被跳过

2. **带分数能见度被分割**
   - `1 1/2SM` 在 tokenize 时被空格分割为 `["1", "1/2SM"]`
   - `is_visibility_token("1")` 返回 `True`（因为 `1.isdigit() and len("1") <= 4`）
   - `1` 被误认为是 1 米能见度，而非 `1 1/2SM` 的整数部分

3. **变化组检测逻辑不完整**
   - `parse_taf()` 中未检测 `FM\d{6}` 格式的变化组

### 修复方案

#### 1. 修复行过滤逻辑 (`src/parser.py:parse_taf()`)

```python
# 修改前：FMddhhmm 格式行被跳过
if not re.match(r'^[A-Z]{4}\s', line):
    if not re.match(r'^(TEMPO|INTER|BECMG|PROB\d{2})\s', line, flags=re.IGNORECASE):
        continue  # 跳过此行

# 修改后：支持 FMddhhmm 格式（6 位数字）
if not re.match(r'^[A-Z]{4}\s', line):
    if not re.match(r'^(TEMPO|INTER|BECMG|PROB\d{2}|FM\d{6})\s', line, flags=re.IGNORECASE):
        continue
```

#### 2. 添加带分数能见度识别 (`src/parser.py`)

```python
def is_visibility_fraction_start(token: str) -> bool:
    """判断是否是能见度带整数部分的开头（如 "1" 在 "1 1/2SM" 中）"""
    # 0-9 的整数可能是带分数能见度的整数部分
    if token.isdigit() and int(token) <= 9:
        return True
    return False
```

#### 3. 修改能见度 token 识别 (`src/parser.py:is_visibility_token()`)

```python
# 修改前：单个数字被误认为是能见度
if token.isdigit() and len(token) <= 4:
    return True

# 修改后：至少 2 位数字才可能是米制能见度
if token.isdigit() and 2 <= len(token) <= 4:
    return True
```

#### 4. 添加带分数能见度解析函数 (`src/parser.py`)

```python
def parse_visibility_with_fraction(int_part: str, frac_part: str) -> Optional[int]:
    """
    解析带整数部分的分数能见度格式（北美报文特有）
    格式：z x/ySM，如 "1" + "1/2SM" = 1 又 1/2 英里 = 1.5 英里
    """
    if not frac_part.endswith('SM'):
        return None
    value_str = frac_part[:-2]  # 去掉 SM
    if '/' not in value_str:
        return None
    num, denom = value_str.split('/')
    int_value = int(int_part)
    frac_value = float(num) / float(denom)
    miles = int_value + frac_value
    return int(miles * 1609.34)
```

#### 5. 修改 parse_weather_state 处理带分数格式

```python
# 在解析标准能见度之前，先检查带分数格式
if not visibility_parsed and is_visibility_fraction_start(token):
    next_token = tokens_list[i + 1] if i + 1 < len(tokens_list) else None
    if next_token and next_token.endswith('SM') and '/' in next_token:
        vis = parse_visibility_with_fraction(token, next_token)
        if vis is not None:
            weather.visibility = vis
            visibility_parsed = True
            i += 2  # 跳过两个 token
            continue
```

#### 6. 修复变化组检测逻辑

```python
# 添加 FMddhhmm 格式检测
elif token.startswith('FM') and len(token) > 2 and token[2:].isdigit():
    is_change_group = True
```

## 2026-03-16: TEMPO 字体显示修复

### 问题描述

在时间线表格中，当同一时段有多个 TEMPO 组时，TEMPO 风列的字体显示过大，影响阅读体验。

**示例报文**:
```
TAF LSGG 140225Z 1403/1509 23012KT 9999 -RA BKN045
TX08/1403Z TN03/1423Z TN01/1506Z
TEMPO 1403/1406 25015G30KT
TEMPO 1403/1410 RA
BECMG 1412/1415 VRB02KT
BECMG 1500/1503 05012KT=
```

**问题表现**:
- TEMPO 列（风、能见度、云、天气）字体明显大于其他列
- 多行 TEMPO 内容显示不统一

### 根本原因

1. **Markdown 换行语法导致 HTML 段落渲染**
   - 原代码使用 `"  \n".join(...)` 连接多个 TEMPO 项
   - `"  "` 是 Markdown 硬换行语法，会被渲染为 `<br>` 或新段落
   - Streamlit 将内容识别为富文本，导致字体渲染异常

2. **`st.table()` 自动换行**
   - 长文本自动换行时字体渲染不一致

### 修复方案

#### 1. 改用纯文本换行 (`app.py`)

```python
# 修改前：使用 Markdown 硬换行语法
main_row["TEMPO 风"] = "  \n".join(tempo_wind)

# 修改后：使用纯文本换行
main_row["TEMPO 风"] = "\n".join(tempo_wind)
```

#### 2. 改用 `st.dataframe()` 替代 `st.table()`

```python
# 修改前
st.table(timeline_data)

# 修改后
st.dataframe(
    timeline_data,
    use_container_width=True,
    hide_index=True,
)
```

#### 3. 添加 CSS 样式统一字体

```python
st.markdown("""
    <style>
    div[data-testid="stDataFrame"] table {
        font-size: 14px !important;
    }
    div[data-testid="stTable"] table {
        font-size: 14px !important;
    }
    div[data-testid="stDataFrame"] td, div[data-testid="stTable"] td {
        font-size: 14px !important;
        line-height: 1.2 !important;
    }
    </style>
""", unsafe_allow_html=True)
```

### 修改文件

- `app.py` - 时间线表格显示逻辑

### 验证结果

```
TEMPO 列字体与其他列保持一致（14px）
多行 TEMPO 内容正常显示，行高紧凑
```

## 2026-03-16: 阵风单位转换和降水强度标记

### 问题描述

解析 TAF 报文 `LSGG 140225Z 1403/1509 23012KT 9999 -RA BKN045 TX08/1403Z TN03/1423Z TN01/1506Z TEMPO 1403/1406 25015G30KT TEMPO 1403/1410 RA BECMG 1412/1415 VRB02KT BECMG 1500/1503 05012KT=` 发现两个问题：

#### 1. 阵风单位转换 (`src/parser.py:parse_wind()`)

```python
    wind.gust = int(gust_match.group(1))

# 修改后：阵风值也按稳定风的计算方法进行单位转换

    gust_value = int(gust_match.group(1))
    if unit == 'KT':
```

#### 2. 降水强度标记 (`src/utils.py:weather_code_to_cn()`)

## 2026-03-16: 短时预报类型统一处理

### 问题描述
TEMPO、INTER、PROB 及其组合格式需要同等对待，时间重叠时取最差值用于航空安全放行。

**支持的短时预报类型**:
- `TEMPO` - 短暂变化
- `INTER` - 间歇性变化
- `PROBxx` - 概率预报（如 PROB30、PROB40）
- `PROBxx TEMPO` - 概率短暂变化
- `PROBxx INTER` - 概率间歇变化
- `INTER PROBxx` - 间歇概率变化

**示例报文**:
```
TAF EDDF 132300Z 1400/1506 19010KT 9999 BKN020
TEMPO 1400/1410 RA BKN010
INTER 1402/1406 -RA BKN015
PROB30 TEMPO 1404/1408 TSRA BKN010CB
PROB40 INTER 1408/1412 SHRA BKN020TCU
```

### 修复方案

#### 1. 添加 INTER 类型识别 (`src/parser.py:parse_taf()`)

```python
# 检查是否是变化组开始
if token in ('FM', 'BECMG', 'TEMPO', 'PROB', 'INTER'):
    pass  # 标准格式
elif token.startswith('PROB') and len(token) > 4 and token[4:].isdigit():
    change_type = 'PROB'  # 规范化为 PROB
elif token.startswith('INTER') and len(token) > 5:
    change_type = 'INTER'
```

#### 2. 支持 INTER 和 PROB 组合 (`src/parser.py:parse_change_group()`)

```python
# 处理 PROBxx 开头（如 PROB30、PROB40）
if first_token.startswith('PROB') and len(first_token) > 4:
    prob_value = int(first_token[4:])
    # 检查下一个 token 是否是 TEMPO 或 INTER
    next_token = next(token_iter)
    if next_token == 'TEMPO':
        change_type = f'PROB{prob_value} TEMPO'
    elif next_token == 'INTER':
        change_type = f'PROB{prob_value} INTER'
```

#### 3. 统一处理短时预报 (`src/parser.py:get_weather_display_at_time()`)

```python
# 收集所有生效的短时预报组
SHORT_TERM_TYPES = ('TEMPO', 'INTER', 'PROB')

for change in taf.changes:
    is_short_term = False
    for st_type in SHORT_TERM_TYPES:
        if st_type in change.type:
            is_short_term = True
            break

    if is_short_term and change.from_time <= query_time < change.to_time:
        short_term_groups.append(change)

# 取最差值
if short_term_groups:
    tempo_weather = _get_worst_tempo(short_term_groups)
```

#### 4. 支持多行报文中的变化组 (`src/parser.py:parse_taf()`)

```python

    if not re.match(r'^(TEMPO|INTER|BECMG|PROB\d{2})\s', line, flags=re.IGNORECASE):

```

## 2026-03-16: PROB 项解析修复

### 问题描述
无法解析含 WMO 报头的 TAF 报文，PROB 项（概率预报）无法正确解析和显示。

**示例报文**:
```
FTDL31 EDZO 132300
TAF EDDF 132300Z 1400/1506 19010KT 9999 BKN020
PROB40 TEMPO 1400/1410 RA BKN010
BECMG 1401/1404 26006KT
BECMG 1408/1412 27011KT
PROB30 TEMPO 1414/1416 SHRA BKN025TCU
BECMG 1416/1419 26004KT CAVOK
PROB30 TEMPO 1500/1506 BKN020=
```

### 根本原因

1. **WMO 报头行被完全跳过**
   - 第一行 `FTDL31 EDZO 132300 TAF EDDF ...` 包含 TAF 标记
   - 但正则匹配逻辑错误，导致整行被跳过

2. **PROB+TEMPO 组合格式未处理**
   - 报文格式：`PROB40 TEMPO 1400/1410 RA BKN010`
   - `PROB40` 和 `TEMPO` 是两个独立 token
   - 解析器将 `TEMPO` 误认为是天气元素而非变化组类型

3. **PROB 项恶劣天气未用于安全放行**
   - PROB 项（30%/40% 概率）通常包含更恶劣的天气
   - 航空安全放行需要显示最坏情况

### 修复方案

#### 1. 修复 WMO 报头行解析 (`src/parser.py:parse_taf()`)

```python
# 修改前：会跳过含 TAF 的 WMO 报头行
if re.match(r'^[A-Z0-9]{4,}\s+[A-Z]{3,4}\s+\d{6}\s*$', line):
    continue

# 修改后：提取 TAF 标记后的内容
if 'TAF' in line.upper():
    taf_match = re.search(r'\bTAF\s+(.*)', line, flags=re.IGNORECASE)
    if taf_match:
        line = taf_match.group(1)  # 提取 TAF 后的内容
    else:
        continue
```

#### 2. 修复 PROB+TEMPO 格式解析 (`src/parser.py:parse_change_group()`)

```python
# 处理 PROB40 TEMPO 格式
if first_token.startswith('PROB') and len(first_token) > 4:
    probability = int(first_token[4:])
    change_type = f'PROB{probability}'

    # 检查下一个 token 是否是 TEMPO
    next_token = next(token_iter)
    if next_token == 'TEMPO':
        change_type = f'PROB{probability} TEMPO'

    # 解析时间
    time_token = next(token_iter)
    from_time, to_time = parse_ddhhddhh(time_token, base_date)
```

#### 3. PROB 项参与最坏情况聚合 (`src/parser.py:get_weather_display_at_time()`)

```python
# 收集所有生效的 TEMPO 和 PROB 组
active_tempo_groups = []
active_prob_groups = []

for change in taf.changes:
    if 'PROB' in change.type:
```

## 2026-03-16: VV 垂直能见度支持

### 问题描述
无法识别 `VVnnn` 格式（垂直能见度），导致云高误判。`VV` 表示垂直垂直能见度（Vertical Visibility），用于当天空被遮蔽物（如雾、烟、尘、火山灰等）完全遮蔽，无法看到云底时，报告从地面到遮蔽物顶部的垂直距离。

**示例报文**:
```
TAF UAAA 132301Z 1400/1424 21002G07MPS 3100 BR FU SCT100
TX07/1409Z TNM07/1401Z
TEMPO 1400/1404 0200 FZFG VV001
BECMG 1405/1407 9999 NSW
BECMG 1414/1415 3100 BR FU
TEMPO 1418/1424 0500 FZFG SCT011=
```

**特殊格式**:
- `VV001` - 垂直能见度 100 英尺
- `VV///` - 垂直能见度不明（遮蔽物顶部高度无法测定）

### 修复方案

#### 1. 添加 VV 识别 (`src/parser.py:is_cloud_token()`)

```python
def is_cloud_token(token: str) -> bool:
    """判断是否是云组"""
    cloud_amounts = ('SKC', 'FEW', 'SCT', 'BKN', 'OVC', 'NCD', 'VV')
    return any(token.startswith(amount) for amount in cloud_amounts)
```

#### 2. 支持 VVnnn 和 VV/// 格式 (`src/parser.py:parse_cloud()`)

```python
def parse_cloud(token: str) -> Optional[Cloud]:
    """解析云组"""
    cloud_amounts = ('SKC', 'FEW', 'SCT', 'BKN', 'OVC', 'VV')

    for amount in cloud_amounts:
        if token.startswith(amount):
            cloud = Cloud(amount=amount)
            remaining = token[len(amount):]

            # 解析云高（对于 VV，这是垂直能见度）
            if remaining and len(remaining) >= 3:
                # 支持 /// 格式（不明）
                if remaining[:3] == '///':
                    cloud.height = None  # 垂直能见度不明
                    remaining = remaining[3:]
                elif remaining[:3].isdigit():
                    cloud.height = int(remaining[:3]) * 100
                    remaining = remaining[3:]

            # 解析云类型
            if remaining in ('CB', 'TCU'):
                cloud.type = remaining

            return cloud

    return None
```

#### 3. VV/// 参与最坏情况聚合 (`src/parser.py:_get_worst_tempo()`)

```python
# 特殊处理 VV/// (垂直能见度不明)
if cloud.amount == 'VV' and cloud.height is None:
    has_vv_unknown = True
    vv_unknown_cloud = deepcopy(cloud)
elif cloud.height is not None:
    # 正常云高处理...

# 优先显示 VV/// (垂直能见度不明是最严重的情况)
if vv_unknown_cloud:
    worst.clouds = [vv_unknown_cloud]

    worst.clouds = [lowest_cloud]
```
